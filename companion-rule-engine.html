Rule-Based Offers + Финансовый Аналитик

Дополнение к ресерчу: конкретные примеры персонализированных предложений из финтех-приложений + полная спецификация rule-based механизма для раздела «Финансовый Аналитик» в мобильном приложении Компаньон

01

Конкретные примеры персонализированных предложений в финтех-приложениях

Revolut — AIR + RevPoints + инсайты

AIR AI-ассистент

«Где мои деньги?» → анализ трат по категориям. «Оцени сбережения» → прогноз накоплений. «Поставь на паузу Spotify» → управление подписками внутри чата. 13 млн UK пользователей, апрель 2026.

RevPoints — геймификация трат

1 RevPoint за £1 потраченный. Бонусы: spare-change, in-app challenges, booking через Revolut Stays. 3000 RevPoints за 150 транзакций новым пользователям. Конвертация в 30+ программ лояльности авиакомпаний.

Персонализированные мультипликаторы

Nespresso ×6, Amazon Prime ×20, GameStop ×17 — мультипликаторы очков за покупки у партнёров. Black Friday: до ×10 RevPoints при оплате через Revolut Pay. Динамические офферы на основе истории трат.

Plan Partnerships

Premium ($9.99/мес): NordVPN, Tinder Plus, Sleep Cycle. Metal ($24.99/мес): + MasterClass, Financial Times, WeWork, ClassPass. Персонализированные подборки на основе категорий трат.

Monzo — Trends + Category Targets + Pots

Trends — единая картина

Объединяет Monzo + внешние счета. Показывает: доход − траты = остаток. Графики неделя-к-неделе, месяц-к-месяцу. Тап по категории → детализация по торговцам. Все клиенты бесплатно (ранее Premium).

Category Targets — лимиты по категориям

Установить месячный лимит на «Еда», «Транспорт», «Развлечения». Система анализирует прошлые траты и предлагает реалистичные цели. Алерт при приближении к лимиту. Plus/Premium: автоматические рекомендации target-ов.

Pots — цифровые конверты

До 20 отдельных «горшочков» для целей: «Отпуск», «Ремонт», «Налоги». Round-up — округление покупок до фунта, сдача в Pot. Virtual cards — отдельная карта на каждый Pot для контроля трат. Bills Manager — автовыделение средств на счета.

Year in Monzo — персонализированные «эры»

Годовой отчёт: «Merchant Era» (топ-торговец), «Travel Era» (разнообразие валют), «Social Era» (Monzo-to-Monzo переводы). Data-driven storytelling. Минимум 15 транзакций для участия. Условные категории на основе паттернов.

Starling — Spaces + Spending Intelligence

Spaces — цифровое кэш-стаффинг

Создание Space для каждой категории: «Бензин», «Продукты», «Праздники». Оплата Direct Debits из конкретного Space. Virtual cards привязаны к Spaces. Round-up — сдача в Space. Savings Goals с процентной ставкой 2.50% AER.

Spending Intelligence (AI-поиск)

«Сколько я потратил на McDonald's за 3 месяца?» → график + аналитика. 50+ категорий трат. Графики и breakdown по ритейлерам. Построено на Google Gemini. Opt-in. Поиск естественным языком по всей истории транзакций.

Starling Assistant (Agentic AI)

Голосовые/текстовые промпты: «Положи £500 на поездку» → автоперевод в Space. «Покажи мой бюджет» → анализ категорий. «Поставь цель накопить £1000 к декабрю» → рассчёт ежемесячных отчислений. Built on Google Gemini/Cloud.

N26 — Insights + Rules + RoundUps

Insights — автокатегоризация

Реальная автокатегоризация транзакций по 20+ категориям. Инстант-нотификации с категорией. Анализ трендов: неделя-к-неделе, месяц-к-месяцу. Подключение внешних банков (Open Banking) для 360° вида.

Rules — автоправила

IF зарплата поступает → перевести 10% в Space «Накопления». IF трата > €50 → округлить до €10, сдачу в Space. IF остаток на счёте > €500 в конце месяца → перевести излишек в Space. Простые условия без ML.

RoundUps + Spaces

Округление до ближайшего евро, сдача в Space. Можно настроить ×2/×5/×10 множитель. Например: покупка на €4.30 → округление до €5.00, €0.70 в Space. С ×5 множителем → €3.50 в Space.

Ally Bank — Buckets + Boosters

Savings Buckets — до 30 целей

Одна Savings Account → до 30 Bucket-ов: «Отпуск», «Аварийный фонд», «Машина». Каждый Bucket: целевая сумма + дата. Все Bucket-ы на одном балансе с процентами. Визуализация прогресса.

Surprise Savings Booster

AI анализирует траты и незаметно переводит «безопасный излишек» в Bucket. Например: обычная трата на кофе $4.50, но на счёте привычный остаток → переводит $5 в Savings. Пользователь не замечает, но копит.

Round-Up Booster

Округление покупок до ближайшего доллара, сдача в Bucket. Можно настроить минимальный порог. Например: не округлять покупки < $5. Средняя экономия: $30-50/мес без усилий.

Spending Buckets

Spending Account → Bucket-ы для регулярных трат: «Аренда», «Коммуналка», «Продукты». Привязка торговцев к Bucket: все траты в Walmart → Bucket «Продукты». Автоматическая категоризация + ручная коррекция.

YNAB — Zero-Based Budgeting (правила, не ML)

Правило 1: Каждому доллару — работа

При поступлении дохода — распределить ВСЕ деньги по категориям до нуля. Нет «свободных» денег. Категории: Giving, Savings, Debt Payments, Recurring Bills, Everyday Expenses. 6-8 мастер-категорий.

Правило 2: Примите реальные расходы

Нерегулярные, но предсказуемые траты → ежемесячные цели. Например: автостраховка $1200/год → $100/мес в категорию. Рождество → $50/мес с января. Нет сюрпризов, нет долгов.

Правило 3: Адаптируйтесь

Перераспределение между категориями при перерасходе. Например: перерасход в «Еде» → перенести из «Развлечений». НЕ наказывает, а учит гибкости. Roll with the punches = главный UX-принцип.

Правило 4: Состарьте деньги

Цель: тратить деньги, которым минимум 30 дней. Избегать paycheck-to-paycheck. Метрика «Age of Money» — средний возраст потраченных денег. > 30 дней = финансовая стабильность.

Ключевой инсайт: Все успешные PFM-системы начинаются с правил и категорий, а не с ML. YNAB ($600 сбережений за 2 месяца, ~$6K за год) построен на 4 простых правилах. Monzo Trends — бесплатно для всех. Starling Spaces — простые цифровые конверты. Правила работают без данных, работают сразу, понятны пользователю.
02

Rule-Based Offer Engine для «Финансового Аналитика»

Концепция

Rule-Based Offer Engine — это слой между категоризацией транзакций и банковскими продуктами. Он не использует ML/AI, а оперирует условиями (IF) на основе данных аналитика: категории трат, суммы, частоты, паттерны, сравнения с прошлым периодом.

Принцип: Каждый совет = триггер (условие) + контекст (данные аналитика) + offer (банковский продукт). Пользователь видит не абстрактный «совет», а конкретное действие с продуктом Компаньона.

Формат правила (Rule Definition)

# rule-definition.yaml — пример правила

rule_id: "transport_to_deposit"
name: "Транспорт → Депозит"
priority: 80
category: "cross_sell"
enabled: true

# Триггер: условия срабатывания
trigger:
  type: "threshold"           # threshold | pattern | comparison | recurring
  conditions:
    - field: "category"
      operator: "in"
      value: ["Транспорт.Общественный.Такси", "Транспорт.Личный.Бензин"]
    - field: "monthly_total"
      operator: ">"
      value: 15000              # 15 000 сомов/мес
      unit: "KGS"
    - field: "period"
      operator: "=="
      value: "current_month"

# Контекст: что показываем пользователю
context:
  insight_template: |
    Вы тратите {{monthly_total}} сом/мес на {{category_name}}.
    За год это {{annual_projection}} сом.
    Если откладывать 20% — {{potential_savings}}/мес.
  data_points:
    - monthly_total
    - annual_projection
    - potential_savings
    - comparison_to_last_month

# Offer: что предлагаем
offer:
  product_id: "deposit_muraz"   # Банковский продукт Компаньона
  product_name: "Депозит «Мураз»"
  offer_template: |
    Открыть депозит «Мураз» на 13% годовых?
    Автопополнение: {{suggested_auto_amount}} сом/мес.
    За год: +{{projected_interest}} сом.
  cta: "Открыть за 1 минуту"
  deeplink: "kompanion://deposits/open?type=muraz&auto={{suggested_auto_amount}}"
  conditions:
    - user.kyc_status == "verified"
    - user.no_active_deposit_of_type("muraz")

# Ограничения показа
constraints:
  max_shows_per_user: 3
  cooldown_days: 14              # Не показывать чаще 1 раза в 14 дней
  time_window: "09:00-20:00"    # Только днём
  channel: ["in_app", "push"]    # Где показывать
  require_min_transactions: 5    # Минимум 5 транзакций в категории

# Метрики для оценки
success_metric:
  type: "conversion"            # conversion | engagement | dismiss
  target: 0.05                  # 5% конверсия = успех
  tracking:
    - event: "offer_shown"
    - event: "offer_tapped"
    - event: "offer_converted"
    - event: "offer_dismissed"

Жизненный цикл правила (Rule Lifecycle)

# rule-lifecycle.yaml

status: draft          # draft → review → active → paused → archived
version: 1
author: "product_team"
reviewers: ["compliance", "risk"]

draft:
  - Написать rule definition
  - Пройти compliance review ( explainability, fairness )
  - Пройти risk review ( financial risk, reputational risk )

review:
  - Staging environment: 100 тестовых пользователей
  - Метрики: show rate, tap rate, conversion rate, dismiss rate
  - Если conversion < target → доработка

active:
  - 10% пользователей → 50% → 100%
  - Ежедневный мониторинг метрик
  - Auto-pause если conversion < 0.5% за 7 дней

paused:
  - Ручная пауза или auto-pause по метрикам
  - Root cause analysis
  - Редактирование и повторный review

archived:
  - Правило устарело или заменено
  - Сохраняется для audit trail

Очередь приоритетов (Offer Ranking)

Когда срабатывает несколько правил одновременно — нужен ranking:

# ranking-algorithm

score = (
  rule.priority * 0.3 +           # Приоритет правила (0-100)
  user.receptivity_score * 0.2 +   # История отклика на эту категорию (0-1)
  product.margin * 0.15 +         # Маржа продукта для банка
  timeliness * 0.2 +              # Актуальность (зарплата = высокая)
  freshness * 0.15                # Новизна (не показывали раньше)
)

# Hard constraints (фильтрация ДО scoring):
# - user.already_has_product → исключить
# - user.kyc_status != "verified" → исключить lending продукты
# - user.cooldown_active(rule_id) → исключить
# - time.now not in rule.time_window → исключить
# - channel not in user.preferred_channels → исключить

# Soft constraints (понижение score):
# - Показывали похожее правило за последние 7 дней → score *= 0.5
# - Пользователь dismiss-ил эту категорию 3+ раз → score *= 0.3

Схема базы данных для Rule Engine

-- Таблица правил
CREATE TABLE rules (
    rule_id VARCHAR(64) PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    priority INT NOT NULL DEFAULT 50,
    category VARCHAR(64),              -- cross_sell | lending | savings | convenience
    trigger_config JSONB NOT NULL,     -- условия срабатывания
    context_config JSONB NOT NULL,     -- шаблоны контекста
    offer_config JSONB NOT NULL,       -- конфигурация offer-а
    constraints_config JSONB NOT NULL, -- ограничения показа
    status VARCHAR(32) DEFAULT 'draft', -- draft | review | active | paused | archived
    version INT DEFAULT 1,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Таблица показов офферов (audit trail)
CREATE TABLE offer_events (
    event_id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    rule_id VARCHAR(64) NOT NULL REFERENCES rules(rule_id),
    event_type VARCHAR(32) NOT NULL,     -- shown | tapped | converted | dismissed | expired
    context_data JSONB,                 -- заменённые переменные в шаблоне
    channel VARCHAR(32),                -- in_app | push | sms
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Индексы для аналитики
CREATE INDEX idx_offer_events_user_rule ON offer_events(user_id, rule_id, event_type);
CREATE INDEX idx_offer_events_created ON offer_events(created_at);
CREATE INDEX idx_rules_status ON rules(status);

-- Materialized view: конверсия по правилам
CREATE MATERIALIZED VIEW rule_conversion_stats AS
SELECT
    rule_id,
    COUNT(*) FILTER (WHERE event_type = 'shown') as shows,
    COUNT(*) FILTER (WHERE event_type = 'tapped') as taps,
    COUNT(*) FILTER (WHERE event_type = 'converted') as conversions,
    ROUND(COUNT(*) FILTER (WHERE event_type = 'converted')::numeric /
          NULLIF(COUNT(*) FILTER (WHERE event_type = 'shown'), 0) * 100, 2) as conversion_rate
FROM offer_events
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY rule_id;

API Rule Engine (псевдокод)

# Pseudo-Python для Rule Evaluation Engine

class RuleEngine:
    def __init__(self, db, user_profile_service):
        self.db = db
        self.profile = user_profile_service
        self.active_rules = self.load_active_rules()

    def on_transaction(self, transaction: Transaction):
        """Главный entry point — вызывается при каждой транзакции"""

        # 1. Категоризация (уже сделано до этого)
        tx = self.enrich_transaction(transaction)

        # 2. Обновить агрегаты пользователя
        self.update_user_aggregates(tx.user_id, tx.category, tx.amount)

        # 3. Оценить все активные правила
        matched = []
        for rule in self.active_rules:
            if self.evaluate_rule(rule, tx):
                matched.append(rule)

        # 4. Rank и фильтрация
        if matched:
            best_offer = self.rank_offers(matched, tx.user_id)
            if best_offer:
                self.deliver_offer(best_offer, tx.user_id, tx)

    def evaluate_rule(self, rule: Rule, tx: Transaction) -> bool:
        """Проверяет ВСЕ условия правила"""
        conditions = rule.trigger_config['conditions']

        for cond in conditions:
            field = cond['field']
            op = cond['operator']
            value = cond['value']

            # Получить значение поля из транзакции или профиля
            actual = self.resolve_field(field, tx)

            if not self.apply_operator(actual, op, value):
                return False  # ANY condition fails → rule doesn't match

        # Все условия прошли → проверить ограничения
        return self.check_constraints(rule, tx.user_id)

    def check_constraints(self, rule: Rule, user_id: int) -> bool:
        """Hard constraints — must ALL pass"""
        c = rule.constraints_config

        # Max shows
        shows = self.db.count(f"""
            SELECT COUNT(*) FROM offer_events
            WHERE user_id = {user_id}
            AND rule_id = '{rule.rule_id}'
            AND event_type = 'shown'
            AND created_at > NOW() - INTERVAL '{c.get('cooldown_days', 30)} days'
        """)
        if shows >= c.get('max_shows_per_user', 999):
            return False

        # Time window
        if c.get('time_window'):
            if not self.in_time_window(c['time_window']):
                return False

        # Product already owned
        if c.get('product_id'):
            if self.profile.has_product(user_id, c['product_id']):
                return False

        # KYC for lending
        if rule.category == 'lending':
            if self.profile.kyc_status(user_id) != 'verified':
                return False

        return True

    def deliver_offer(self, rule: Rule, user_id: int, tx: Transaction):
        """Доставка offer-а в нужный канал"""
        context = self.build_context(rule, tx)
        channels = rule.constraints_config.get('channel', ['in_app'])

        for channel in channels:
            if channel == 'push' and self.profile.push_enabled(user_id):
                self.send_push(rule, user_id, context)
            elif channel == 'in_app':
                self.queue_inapp_card(rule, user_id, context)

        # Логирование
        self.log_event(rule.rule_id, user_id, 'shown', context, channel)
